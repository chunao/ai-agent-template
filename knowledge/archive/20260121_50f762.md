# Claude Code Skills、結局どれを入れる？用途別おすすめ9選

## TL;DR & Key Conclusions
この記事では、Claude CodeのSkillsの中から実用的な9つを厳選し、用途別に分類して紹介しています。Skillsを入れたいが何を選ぶべきか迷っている開発者向けに、プロジェクト開発と普段使いのシーンに応じた具体的なユースケースと使用例を提示します。

- **Playwright Browser Automation**: ブラウザ自動化・E2Eテスト・UIテストに最適。カスタムコードをその場で記述・実行可能
- **prompt-engineering**: LLMプロンプトの最適化。カスタムスキル作成やサブエージェント設計に不可欠
- **software-architecture**: Clean ArchitectureやSOLID原則を適用したアーキテクチャ設計を支援
- **subagent-driven-development**: 個々のタスクに独立したサブエージェントを派遣し、2段階のレビューを実施
- **Kaizen**: 継続的改善の哲学をソフトウェア開発に適用。プロセスとコードの段階的改善を支援
- **brainstorming**: ラフなアイデアを対話と質問を通じて構造化された設計に変換
- **Canvas Design**: デザイン哲学を基にした美しいビジュアルアート（PNG/PDF）を作成
- **Image Enhancer**: 画像やスクリーンショットの解像度・品質を向上させる
- **ship-learn-next**: フィードバックループに基づき、次に何を構築・学習すべきかを決定

## Quickstart
1. **リポジトリの確認**: [awesome-claude-skills](https://github.com/ComposioHQ/awesome-claude-skills)にアクセス
2. **用途の特定**: プロジェクト開発向け（5選）か普段使い向け（6選）かを判断
3. **Skillの選択**: 自分のワークフローに合ったものを1〜2個選択
4. **インストール**: 各SkillのREADMEに従ってClaude Codeに追加
5. **試用**: 記事に記載された使用例を参考に実際に試す

## Context & Claims (Claim-Evidence-Caveat)
- **Claim**: Claude CodeのSkillsは豊富だが、どれを選ぶべきかわかりにくい
    - **Evidence**: awesome-claude-skillsリポジトリには多数のSkillsが存在するが、各Skillの具体的なユースケースや選び方の指針が不足
    - **Caveat**: 本記事では9個のSkillに絞り込んでいるため、全てのSkillsをカバーしているわけではない

- **Claim**: Playwright Browser Automationはブラウザ自動化に最適
    - **Evidence**: ログインフローのテスト、フォーム送信の自動化、レスポンシブデザイン検証などの具体例を提示
    - **Caveat**: 事前知識としてPlaywrightの基本的な使い方を理解している必要がある場合がある

- **Claim**: prompt-engineeringはプロジェクト・普段使いの両方で有用
    - **Evidence**: カスタムスキル作成、サブエージェント設計、プロンプト最適化など幅広いユースケース
    - **Caveat**: Anthropicのベストプラクティスを学ぶ必要があり、初心者には学習コストがかかる

- **Claim**: subagent-driven-developmentは品質を保ちながら並行開発を実現
    - **Evidence**: 実装→仕様準拠チェック→コード品質チェックの2段階レビュープロセス
    - **Caveat**: 実装計画が必要であり、タスクが明確に定義されていないと効果が薄い

- **Claim**: Kaizenは継続的改善の哲学をソフトウェア開発に適用
    - **Evidence**: プロセスのボトルネック分析、段階的リファクタリング、技術的負債の解消
    - **Caveat**: 「小さな改善の積み重ね」という考え方が、即座の大きな成果を求める場合には合わない

- **Claim**: brainstormingは要件整理と設計の前段階で有効
    - **Evidence**: 一度に一つの質問、選択式優先、2〜3のアプローチを提案してから決定
    - **Caveat**: 対話を通じた設計プロセスが好まれない場合や、急いでいる場合には冗長に感じる可能性

- **Claim**: Canvas Designはビジュアルデザインを2段階プロセスで作成
    - **Evidence**: デザイン哲学を作成後、それを視覚化。90%ビジュアル、10%テキスト
    - **Caveat**: デザインの品質はClaude Codeの画像生成能力に依存し、プロのデザインツールには及ばない可能性

- **Claim**: Image Enhancerは画像品質を向上させる
    - **Evidence**: スクリーンショットの解像度アップスケール、エッジのシャープ化、テキストの明瞭度向上
    - **Caveat**: 元画像の品質が極端に低い場合、改善には限界がある

- **Claim**: ship-learn-nextはデータ駆動の意思決定を支援
    - **Evidence**: 小さな実験→フィードバック収集→次のステップ決定のサイクル
    - **Caveat**: フィードバックの収集と分析に時間がかかり、迅速な意思決定が必要な場合には不向き

## Normalized Conditions
- **Use Cases**:
    - ブラウザ自動化やE2Eテストを実施したい場合（Playwright Browser Automation）
    - カスタムスキルやサブエージェント用プロンプトを作成・最適化したい場合（prompt-engineering）
    - Clean ArchitectureやSOLID原則に基づいたアーキテクチャ設計を行いたい場合（software-architecture）
    - 実装計画に基づいて品質チェックを挟みながら開発を進めたい場合（subagent-driven-development）
    - プロセスやコードを継続的に改善したい場合（Kaizen）
    - 新機能やプロジェクトのアイデアを具体的な設計に落とし込みたい場合（brainstorming）
    - イベントポスターやブランドアートを作成したい場合（Canvas Design）
    - スクリーンショットや画像の品質を向上させたい場合（Image Enhancer）
    - 次に学ぶべき技術や構築すべき機能を優先順位付けしたい場合（ship-learn-next）
- **Anti-Cases**:
    - 単純なタスクやすぐに完了する作業（多くのSkillsではオーバーヘッドが大きすぎる）
    - Playwriteの知識がなく、ブラウザ自動化の必要性がない場合（Playwright Browser Automation）
    - プロンプトエンジニアリングの学習に時間をかけたくない場合（prompt-engineering）
    - 小規模なプロトタイプや実験的なコード（software-architecture）
    - 実装計画が不明確な、または一人で小規模な開発を行う場合（subagent-driven-development）
    - 大規模な一括変更が必要で、小さな改善では対応できない場合（Kaizen）
    - すでに明確な設計があり、対話的な要件整理が不要な場合（brainstorming）
    - プロフェッショナルなデザインツール（Figma、Photoshop等）が必要な場合（Canvas Design）
    - 元画像の品質が極端に低い、またはそもそも画像改善が不要な場合（Image Enhancer）
    - 直感やビジョンで迅速に意思決定したい場合（ship-learn-next）
- **Prerequisites**:
    - Claude Codeがインストールされている環境
    - awesome-claude-skillsリポジトリへのアクセス
    - 各Skillの説明やREADMEを読む時間
    - Playwright Browser Automation: Playwrightの基本理解（推奨）
    - prompt-engineering: Anthropicのベストプラクティスへの関心
    - software-architecture: Clean ArchitectureやSOLID原則の基本知識（推奨）
    - subagent-driven-development: 実装計画の存在
    - Kaizen: 継続的改善の考え方に対する理解
    - brainstorming: 対話を通じた設計プロセスへの忍耐
    - Canvas Design/Image Enhancer: Claude Codeの画像生成・編集機能
- **Decision Triggers**:
    - ブラウザテストの必要性が発生（Playwright Browser Automation）
    - カスタムスキルを作成する必要が出た（prompt-engineering）
    - プロジェクトのアーキテクチャ設計フェーズに入った（software-architecture）
    - 複数のタスクを並行して品質を保ちながら実行したい（subagent-driven-development）
    - プロセスやコードに改善の余地を感じた（Kaizen）
    - 新機能のアイデアを形にする前に要件を整理したい（brainstorming）
    - イベントやブランディングのためのビジュアル素材が必要（Canvas Design）
    - ブログや資料用のスクリーンショットの品質が低い（Image Enhancer）
    - 学習や開発の次のステップに迷った（ship-learn-next）
- **Failure Modes / Risks**:
    - Skillの数が多すぎると、どれを使うべきか混乱する
    - 学習コストが高いSkillは、初期段階で挫折する可能性
    - 実装計画が不明確な場合、subagent-driven-developmentは効果が薄い
    - デザイン品質がプロのツールには及ばない可能性（Canvas Design）
    - 画像品質の改善には限界がある（Image Enhancer）
    - フィードバック収集に時間がかかり、意思決定が遅れる可能性（ship-learn-next）
- **Operational Cost**:
    - 各Skillのインストールと初期設定に数分〜数十分
    - prompt-engineeringやsoftware-architectureは学習コストが高い
    - subagent-driven-developmentは実装計画の作成コストが必要
    - brainstormingは対話プロセスに時間がかかる
    - ship-learn-nextはフィードバック収集と分析に時間を要する
