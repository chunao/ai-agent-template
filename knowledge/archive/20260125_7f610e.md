# 2025年12月時点の「ぼくがいまやっている最強の開発方法」

## TL;DR & Key Conclusions
OpenAI の Codex（GPT-5.2-Codex）を使用し、要件定義・設計・実装・コードレビューを自動化する開発ワークフローを確立。特に「独立したコンテキストでのコードレビュー」により、バグを削減し開発速度を向上させている。

- **Codexへの移行**: 2025年初期のClaude Code → 8月にSpec Driven → 11月にCodexに移行
- **Codexの特長**: 高いコーディング性能と淡白なコミュニケーション
- **最大の効果**: `my-review`スクリプトによる自動化されたコードレビュー
- **独立コンテキストレビュー**: 同じモデルで書いたコードでも指摘が返ってくる
- **バグ削減**: リリース後の手戻り減少を実現
- **reasoning_effort設定**: xhigh設定が必須（mediumでは破綻しやすい）
- **人間に迎合しない**: 反論したり意見を貫く傾向で設計精度が向上
- **トレードオフ**: 10分単位で完了時間が延びる
- **週間容量**: Codex使用時は20～40%が残る
- **継続的最適化**: 3ヶ月ごとに手法を更新している傾向
- **コード品質**: 全体整合性が高く、重複が少ない
- **指示忘れリスク**: ストレートではないやり取りで承認取得ステップを忘れることがある

## Quickstart

### ワークフロー構成（5ステップ）

```
1. 要件定義・設計ドキュメント作成（docs/dev/{feature_name}/）
2. 実装フェーズ
3. lint/build/test実行
4. my-reviewコマンドでコードレビュー（タイムアウト：900秒）
5. 改善点反映→レビュー反復（最大2回）
```

### my-review スクリプト

```bash
#!/bin/bash
exec codex exec -c model_reasoning_effort=high 2>/dev/null "$@"
```

### Codex設定ファイル (~/.codex/config.toml)

設定項目：
- モデル：gpt-5.2-codex
- reasoning_effort：xhigh
- sandbox_mode：workspace-write
- approval_policy：on-failure

### Slackスクリプト例

```bash
#!/bin/bash
MSG=$(echo $@ | tr -d '"=')
exec curl -X POST --data-urlencode \
  "payload={\"text\":\"message: ${MSG}\"}" \
  https://hooks.slack.com/services/T00XXX00/BDXXXXXXXX/XXXXXXXXXXXXXXXXXXXX
```

### プロンプトテンプレート

定型7ステップフロー：
1. feature名決定
2. docs作成
3. 開発
4. lint/build/test
5. my-review実行
6. 改善反復（最大2回）
7. Slack通知

スニペット管理ツールで再利用可能。

## Context & Claims (Claim-Evidence-Caveat)

- **Claim**: 独立したコンテキストでのコードレビューが最も効果的
    - **Evidence**: 「同じモデルで書いたコードでも指摘が返ってくるので、独立したコンテキストで実施するのが良い」。バグ減少とリリース後の手戻り減少を報告
    - **Caveat**: 10分単位で完了時間が延びるトレードオフあり

- **Claim**: Codexはsonnet/opus-4.5より優れている
    - **Evidence**: コード重複が少ない、全体整合性が高い、コード生成に特化。Codex使用時は週間容量の20～40%が残る
    - **Caveat**: コーディング性能は「reasoning_effort=xhigh」設定が必須。mediumでは破綻しやすい

- **Claim**: Codexは人間に迎合しない
    - **Evidence**: opus-4.5と異なり、反論したり意見を貫く傾向。設計精度が向上
    - **Caveat**: 指示忘れが発生することがあり（例：承認取得ステップ）、ストレートではないやり取りで顕著

- **Claim**: 継続的な手法の最適化が重要
    - **Evidence**: 著者は3ヶ月ごとに手法を更新。「たぶん3ヶ月後には違うやり方になっている」と述べている
    - **Caveat**: 頻繁な変更により学習コストが発生する可能性

## Normalized Conditions

- **Use Cases**:
    - 多レイヤーアプリケーション（Client-Server-DB構成）
    - TypeScript/Python混在開発
    - MCP Server on Cloud Run、バッチ処理など複数技術スタック
    - 手戻りを削減したい
    - バグを事前に検出したい
    - コード品質の均一化が必要
    - 開発速度より品質重視

- **Anti-Cases**:
    - 数ファイル以内で完結するスクリプト（単一言語でシンプル）
    - 日本語/英語指示だけで十分な小規模タスク
    - 即座の開発スピードが最優先

- **Decision Triggers**:
    - 手戻りを削減したい
    - バグを事前に検出したい
    - コード品質の均一化が必要
    - 開発速度より品質重視
    - 複数技術スタックを扱う必要がある
