# Claude Code で GitHub Projects管理効率化

## TL;DR & Key Conclusions
AI Coding Agentの進化により、コード実装は高速化したが、Issue作成やレビューに時間を要する課題が発生。この記事は、Claude Codeのカスタムコマンド定義により、GitHub Projects管理を自動化する実践例を提示している。

- **課題**: AI実装の高速化に伴い、Issue作成とプロジェクト設定（Epic/Feature/Task紐付け、Sprint設定、Status管理）が開発のボトルネックになった
- **解決策**: `/create-issue` コマンドを定義し、Issue作成からプロジェクトフィールド設定までを一括自動化
- **技術要点**: GraphQL APIとNode ID形式を使用することで、GitHub Projects V2の操作を確実に実行できる
- **効率化**: 複数フィールドの並列設定により処理時間を短縮。親Issue分析を事前実施し子Issue内容の精度を向上
- **API理解**: Issue番号（REST API）、Issue Node ID（GraphQL）、Project ID（GraphQL）、Project Item ID（Project内実体）の4つのID体系を正確に理解する必要がある
- **注意点**: 新規Issue作成直後は索引化に遅延の可能性。Project Item IDは毎回動的に取得が必要
- **運用ルール**: Epic > Feature > Task/Bugの階層構造。ステータスはReady→Todo→In Progress→In Review→Doneのフロー

## Quickstart
1. **Claude Codeのカスタムコマンド定義**: `.claude/commands/create-issue.md` を作成し、Issue作成ロジックを記述
2. **gh CLIコマンドの構築**: `gh issue create` でIssue作成、`gh api graphql` でプロジェクト設定を実行
3. **親Issue分析の実装**: `--parent` フラグ指定時に親Issueを事前取得・分析し、子Issue内容を自動生成
4. **フィールド設定の並列化**: GraphQL mutationで複数フィールドを同時設定し処理時間を短縮
5. **コマンド実行**: `/create-issue --parent=11111 --next --task 親issueに対する実装タスクを作成`

## Context & Claims (Claim-Evidence-Caveat)
- **Claim**: AI実装の高速化により、Issue管理がボトルネックになる
    - **Evidence**: 「AI Coding Agentによって、簡単なタスクであれば素早く実装ができるようになってきた」「Issueの作成やGitHub Projectsの設定に手間がかかる」
    - **Caveat**: 組織やプロジェクトの規模、運用ルールによって課題の深刻度は異なる

- **Claim**: GraphQL APIとNode ID形式を使用することで、GitHub Projects V2操作を確実に実行できる
    - **Evidence**: 「GitHub Projects V2 の操作には GraphQL API を使う必要がある」「Issue Node ID、Project ID、Project Item ID の3つのIDが必要」
    - **Caveat**: REST APIとGraphQL APIで異なるID形式を扱うため、API体系の正確な理解が必須

- **Claim**: 親Issue分析により子Issue内容の精度が向上する
    - **Evidence**: 「親Issueを事前に取得・分析することで、子Issueの内容をより適切に生成できる」
    - **Caveat**: 親Issueの記述が不十分な場合、分析結果の品質も低下する

- **Claim**: 複数フィールドの並列設定で処理時間を短縮できる
    - **Evidence**: GraphQL mutationで複数フィールドを同時設定する実装例
    - **Caveat**: 並列設定可能なフィールド数やAPI制限に注意が必要

- **Claim**: Epic > Feature > Task/Bugの階層構造で要件管理を整理できる
    - **Evidence**: 「Epic: 複数のFeatureをまとめる」「Feature: PdMが管理する要件定義」「Task: エンジニアが実装する1週間スプリント単位の作業」
    - **Caveat**: チーム規模や開発手法により、階層構造の適合度は異なる

## Code Snippets
- **基本的なIssue作成コマンド**: `gh issue create` でタイトル・本文・ラベルを指定してIssueを作成
- **プロジェクトフィールド設定（GraphQL）**: `gh api graphql` でProject Item IDを取得し、複数フィールドを並列設定
- **親Issue分析ロジック**: `--parent` フラグ指定時に `gh issue view` で親Issueを取得・分析
- **カスタムコマンド定義**: `/create-issue --parent=${issue_id} --next --task` などのフラグでIssue作成を制御

## Normalized Conditions
- **Use Cases**:
    - AI Coding Agentを活用した高速開発で、Issue管理が追いつかない場合
    - GitHub Projectsで複雑な階層構造（Epic/Feature/Task）を運用している場合
    - Sprint管理やステータスフローを厳密に運用するチーム開発
    - Issue作成時のプロジェクトフィールド設定を自動化したい場合
    - 親Issueから子Issueを大量に派生させる必要がある場合
- **Anti-Cases**:
    - GitHub Projectsを使用していない、または簡易的にしか使っていない場合
    - Issue管理が複雑でなく、手動設定で十分な小規模プロジェクト
    - Claude Codeやgh CLIを使用していない環境
    - GraphQL APIの学習コストを許容できない場合
- **Decision Triggers**:
    - Issue作成とプロジェクト設定に1件あたり3分以上かかるようになった
    - AI実装の速度にIssue管理が追いつかなくなった
    - Sprint計画時に大量のTaskを一括作成する必要が生じた
    - 親Issue分析に基づく子Issue自動生成で開発精度を向上させたい
