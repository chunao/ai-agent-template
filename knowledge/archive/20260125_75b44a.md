# いつの間にか GitHub issue で依存関係を設定できるようになってる！

## TL;DR & Key Conclusions
GitHub IssueにIssue間の依存関係を明示的に管理する機能が追加されました。従来はコメント履歴に埋もれていた先行・後続関係を、サイドバーから設定し、GitHub Projectsのテーブルビューで可視化できます。複数Issueを管理する際の認識負荷が大幅に軽減されます。

- **2つの関係性タイプ**: `Is blocking`（後続Issueの着手をブロック）と `Blocked by`（先行Issueによってブロックされている）を設定可能
- **双方向同期**: Issue #111で #123をblockingに設定すると、自動的にIssue #123では #111がblocked byとして表示される
- **テーブルビュー対応**: GitHub Projectsのテーブルビューで先行Issue未解決時に赤いマークが表示され、一目で状態確認可能
- **ビュー制限**: ロードマップビューやカンバンビューでは依存関係が表示されない（2026年1月時点）
- **設定方法**: Issue右側のサイドバー「Relationships」セクションから直感的に追加・変更可能
- **複数関係対応**: 1つのIssueに複数の依存関係を設定でき、「View all →」でリスト表示
- **認識負荷軽減**: コメント履歴を何度も確認する必要がなくなり、効率的なタスク管理が実現
- **中規模以上向け**: Issue数が増加したプロジェクトで真価を発揮する機能
- **機能拡張待ち**: ガントチャート形式の矢印コネクタなど、より複雑な依存関係表現は今後に期待

## Quickstart
1. **Issue画面を開く**: 依存関係を設定したいIssueのページにアクセス
2. **サイドバーから設定**: 右側の「Relationships」セクションを確認
3. **先行関係を追加**: 「Add or change blocked by」をクリックして、着手を妨げている先行Issueを追加
4. **後続関係を追加**: 「Add or change blocking」をクリックして、当該Issueが妨げている後続Issueを追加
5. **Projects テーブルビューで確認**: GitHub Projectsのテーブルビューで、先行Issue未解決時の赤いマークを確認
6. **複数設定時**: 複数の依存関係がある場合は「View all →」でリスト表示を活用

## Context & Claims (Claim-Evidence-Caveat)
- **Claim**: GitHub Issueの依存関係管理機能は、複数Issue管理における認識負荷を大幅に軽減する
    - **Evidence**: 記事著者の経験として「従来は依存関係がコメント履歴に埋もれ、何度も確認する非効率性があった」と指摘。テーブルビューで先行Issue未解決時に赤いマークが表示され、「いちいち開かなくて済んでとても嬉しい」と評価
    - **Caveat**: 可視化対応はテーブルビューのみ。ロードマップビューやカンバンビューでは依存関係が表示されない（2026年1月時点）

- **Claim**: 双方向同期により、依存関係の整合性が自動的に保たれる
    - **Evidence**: Issue #111で #123をblockingに設定すると、自動的にIssue #123では #111がblocked byとして表示される相互参照機能が実装されている
    - **Caveat**: 手動でのコメント管理と異なり、システムが自動管理するため、誤設定時の修正もUI経由で行う必要がある

- **Claim**: UIベースの設定により、技術的な知識がなくてもチーム全体で活用可能
    - **Evidence**: Issue右側のサイドバー「Relationships」セクションから直感的に追加・変更が可能な設計
    - **Caveat**: GitHub Projectsのテーブルビューを利用しない場合、可視化のメリットを享受できない

## Normalized Conditions
- **Use Cases**:
    - 複数のIssueを管理する中規模以上のプロジェクト
    - 機能開発において前提条件のある作業フロー（例: APIエンドポイント実装 → フロントエンド統合）
    - テーブルビューでタスク進捗を可視化したい場面
    - GitHub Projectsを既に活用している組織
    - Issue数が増加し、依存関係追跡の重要性が高まった段階
- **Anti-Cases**:
    - ロードマップビューやカンバンビューでの依存関係可視化が必須の場合
    - ガントチャート形式の矢印コネクタが必要な複雑なプロジェクト管理
    - Issue数が少なく（10件未満）、依存関係が単純なプロジェクト
    - GitHub Projectsを使用しないチーム（可視化のメリットを享受できない）
- **Decision Triggers**:
    - Issue間の依存関係がコメント履歴に埋もれ、管理が煩雑になってきた時
    - テーブルビューでの管理が実務フローに適合している場合
    - チームメンバーが「このIssueの前提条件は？」と頻繁に確認する状況が発生した時
    - プロジェクト規模が拡大し、Issue数が50件を超えた段階
