# Claude Code × Obsidian Daily Note運用を半年続けて3つの改善を試してみた

## TL;DR & Key Conclusions
Claude CodeとObsidianを用いたDaily Note運用を半年間継続し、振り返りの形骸化、質問の手間、タスク状態の視覚化という3つの課題を特定。それぞれに対してdaily-eveningコマンドの改善、monthly-reviewコマンドの実装、CSS Snippetsによるタスクステータスの色分けという解決策を導入した。

- **振り返りの形骸化への対応**: 日々の振り返りを目標に沿ったKPT形式（Keep・Problem・Try）に変更し、成長目標との連動を強化
- **質問の手間の削減**: AskUserQuestionツールを使用したTab付きインタラクティブなチェックボックス形式に変更し、タスクステータスの選択を効率化
- **monthly-reviewコマンドの実装**: 月次でデイリーノートを自動集計し、案件別実績、目標別KPT分析、定量的実績を生成
- **タスク状態の視覚化**: CSS Snippetsで未着手（白）、進行中（オレンジ）、レビュー中（青）、完了（緑）を色分け表示
- **Google Calendar連携**: daily-morningで今日の予定取得、daily-eveningで実施確認を自動化
- **タスク自動繰越**: 未完了タスクを自動的に「明日やる」セクションに転記

## Quickstart
1. **環境変数の設定**: PROJECT_A、PROJECT_B、PROJECT_C、GOAL_1、GOAL_2、GOAL_3を定義
2. **daily-morningの実行**: 前日の「明日やる」とGoogle Calendar予定から今日のDaily Noteを生成
3. **daily-eveningの実行**: タスクステータスをチェックボックスで選択し、目標別の振り返りを入力してDaily Noteを更新
4. **monthly-reviewの実行**: 引数なし（実行月）または`YYYY-MM`形式で指定月の振り返りファイルを生成
5. **CSS Snippetsの有効化**: Obsidianの設定からCSS snippetsでcustom-task-statusを有効化

## Context & Claims (Claim-Evidence-Caveat)

- **Claim**: 日々の振り返りは成長目標と連動させないと形骸化する
    - **Evidence**: 運用開始2ヶ月後、「今日は頑張ってタスクできた、集中してできた、疲れた、特になし」といった適当な振り返りになっていた
    - **Caveat**: 目標に沿ったKPT形式に変更することで、より構造化された振り返りが可能になるが、ユーザーが真摯に振り返りを行う必要がある

- **Claim**: シーケンシャルな質問形式は時間効率が悪い
    - **Evidence**: 6つの質問に順次回答する必要があり、案件Aに3つのタスクがあれば一つの質問で全て音声または手動入力しなければならなかった
    - **Caveat**: AskUserQuestionツールでTab付きインタラクティブな選択形式にすることで効率化できるが、全てのプロジェクトで同じステータス体系が使える場合に最適

- **Claim**: ObsidianではMarkdownのカスタムチェックボックスが全て同じ見た目になる
    - **Evidence**: `- [/]` や `- [R]` などの記号は、VS Codeでは区別できるが、Obsidianでは全てチェックボックスとして表示され進捗状態が視覚的に区別できなかった
    - **Caveat**: CSS Snippetsで色分けすることで視覚化できるが、Obsidianで設定を有効化する必要がある

- **Claim**: 月次振り返りを手動で集計するのは非効率
    - **Evidence**: デイリーノートを1日1日手動で確認して集計する作業は時間がかかり、見落としのリスクがある
    - **Caveat**: monthly-reviewコマンドで自動化できるが、デイリーノートが一定のフォーマットに従っている必要がある

- **Claim**: タスクステータスは限られたパターンで管理できる
    - **Evidence**: 実運用では「未着手、進行中、レビュー中、完了、中止」の5ステータスで十分であることが判明
    - **Caveat**: プロジェクトや組織によってはより詳細なステータス管理が必要な場合もある

## Code Snippets

- **daily-eveningコマンド**: Google Calendarイベントを取得し、デイリーノートからタスクを読み込んで、AskUserQuestionツールでタスクステータスを選択形式で質問。目標別の振り返りを入力してDaily Noteを更新
- **monthly-reviewコマンド**: 引数なしまたは`YYYY-MM`形式で実行し、対象期間のデイリーノートを検索・読み込み、案件別実績、目標別KPT分析、定量的実績を含む月次振り返りファイルを生成
- **CSS Snippets（タスクステータスの色分け）**: `[/]`をオレンジ、`[R]`を青、`[x]`を緑でタスクチェックボックスを色分け表示するCSSコード
- **daily-morningコマンド**: 前日のDaily Noteから「明日やる」を取得し、Google Calendarの予定を読み込んで今日のDaily Noteを作成
- **タスク自動繰越ロジック**: 「今日のTodo」から未完了タスク（`[ ]`、`[/]`、`[R]`）を抽出し、完了（`[x]`）と中止（打ち消し線）を除外して「明日やる」に転記

## Normalized Conditions

- **Use Cases**:
    - ObsidianとClaude Codeを使用した個人の日次タスク管理
    - Google Calendarと連携したタスク管理
    - 複数プロジェクトを並行して進めるエンジニアの進捗管理
    - 成長目標に基づいた定期的な振り返り習慣の確立
    - 月次での成果と課題の可視化
    - デイリーノートの自動化とフォーマット統一

- **Anti-Cases**:
    - Obsidianを使用しない環境
    - Claude Codeが利用できない環境
    - チーム全体でのタスク管理（個人利用に最適化されている）
    - 非常に詳細なステータス管理が必要な場合
    - Google Calendarを使用しない環境

- **Prerequisites**:
    - Obsidian（Daily Noteプラグイン）
    - Claude Code CLI
    - Python環境（uvパッケージマネージャ）
    - Google Calendar API（today-calendar.pyスクリプト）
    - 環境変数の設定（PROJECT_A/B/C、GOAL_1/2/3）
    - Bashコマンド実行環境
    - Obsidianの`.obsidian/snippets/`ディレクトリにCSSファイル配置

- **Decision Triggers**:
    - Daily Noteの振り返りが形骸化してきたと感じた時
    - タスクステータスの入力に時間がかかっていると感じた時
    - Obsidianでタスクの進捗状態が見づらいと感じた時
    - 月次の振り返りを効率化したいと考えた時
    - 複数プロジェクトのタスク管理を効率化したいと考えた時
    - 成長目標との紐づけを強化したいと考えた時
