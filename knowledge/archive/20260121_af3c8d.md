# Claude CodeでGitHub Issue駆動開発を自動化する

## TL;DR & Key Conclusions
Claude CodeとIssue Workflowツールキットを組み合わせることで、GitHub Issue駆動開発の全フロー（Issue読み込み→ブランチ作成→TDD実装→品質チェック→PR作成→レビュー→マージ）を自動化できます。公式プラグインとの連携により、開発者は手動作業を最小限に抑え、コード品質を保ちながら効率的に開発を進められます。

- **TDD強制ワークフロー**: Red-Green-Refactorサイクルを遵守し、テスト駆動開発を自動的に実施
- **品質ゲートの自動化**: lint、format、typecheckを自動実行し、コード品質を担保
- **Issue進捗自動報告**: 開発の進行状況をGitHub Issueに自動的に報告
- **公式プラグイン連携**: Claude Code公式プラグイン（commit-push-pr、pr-review-toolkit）と組み合わせて完全なワークフローを実現
- **並行作業サポート**: git worktreeを活用し、複数のIssueを同時並行で進められる環境を提供
- **自動起動スキル**: 実装中に条件に応じて自動的にトリガーされるスキルが品質を維持
- **カスタマイズ可能**: workflow-config.jsonで品質ツールやワークフローをプロジェクトに合わせて調整可能

## Quickstart
1. **uvのインストール**: まず[uv](https://docs.astral.sh/uv/getting-started/installation/)をインストール
2. **Issue Workflowのインストール**: `uv tool install git+https://github.com/drillan/issue-workflow.git`を実行
3. **プロジェクト初期化**: `issue-workflow init --language python`でプロジェクトに設定ファイルとスキルファイルを生成
4. **公式プラグインのインストール**: Claude Code上で`/plugin`コマンドを実行
5. **Issue開始**: `/start-issue <番号>`でIssueベースの開発を開始
6. **実装とテスト**: Claude CodeがTDDサイクルに従って実装を進める
7. **PR作成**: `/commit-push-pr`でコミット、プッシュ、PR作成を一括実行
8. **レビュー**: `/pr-review-toolkit:review-pr`でPRをレビュー
9. **レビュー対応**: `/pr-review-toolkit:review-pr-comments`でレビューコメントに対応
10. **マージ**: `/merge-pr <番号>`でPRをマージ

## Context & Claims (Claim-Evidence-Caveat)
- **Claim**: Issue Workflowは開発フロー全体を自動化し、手動作業を削減する
    - **Evidence**: Issue読み込み、ブランチ作成、TDD実装、品質チェック、PR作成、レビュー、マージまでの一連の流れをコマンドで実行可能
    - **Caveat**: 初期セットアップとして.claude/ディレクトリへの設定ファイル配置が必要

- **Claim**: TDD強制ワークフローによりテスト駆動開発が確実に実施される
    - **Evidence**: Red-Green-Refactorサイクルの遵守を自動化
    - **Caveat**: workflow-config.jsonでtdd_required設定をtrueにする必要がある

- **Claim**: 品質ゲートにより一定のコード品質が保証される
    - **Evidence**: lint、format、typecheckを自動実行してPR前に品質チェック
    - **Caveat**: 各品質ツールのコマンドはworkflow-config.jsonで適切に設定する必要がある

- **Claim**: git worktreeを使った並行作業により、複数Issueの同時進行が可能
    - **Evidence**: `/add-worktree`コマンドでIssue単位で独立した作業環境を作成
    - **Caveat**: ワークツリーは親ディレクトリに配置されるため、ディスク容量に注意

- **Claim**: GitHub Actionsとの統合により、PRレビューを完全自動化できる
    - **Evidence**: PR作成時に自動でレビューが実行され、結果がコメントとして投稿
    - **Caveat**: GitHub Actionsの設定が別途必要（関連記事を参照）

## Code Snippets
- **プロジェクト初期化コマンド**: uvを使ってIssue Workflowをインストールし、プロジェクトに必要な設定ファイルを生成
- **品質設定JSON**: .claude/workflow-config.jsonで品質ツール（lint、format、typecheck、test）とワークフロー設定を定義
- **主要ワークフローコマンド**: /start-issue、/commit-push-pr、/pr-review-toolkit:review-pr、/pr-review-toolkit:review-pr-comments、/merge-prなど、開発フロー全体をカバーするスラッシュコマンド群

## Normalized Conditions
- **Use Cases**:
    - GitHub Issue駆動開発を実践しているチーム開発プロジェクト
    - TDD（テスト駆動開発）を強制したいプロジェクト
    - コード品質チェックを自動化し、レビュー負荷を軽減したい場合
    - 複数のIssueを並行して進める必要がある開発環境
    - Claude Codeを開発ツールとして採用しているプロジェクト
- **Anti-Cases**:
    - GitHub Issueを使用しない小規模な個人プロジェクト
    - 探索的プロトタイピングや実験的なコーディング
    - テストコードを書かない開発方針のプロジェクト
    - Claude Code以外の開発環境を使用している場合
- **Prerequisites**:
    - uvパッケージマネージャーがインストールされていること
    - GitHubリポジトリとIssueが存在すること
    - Claude Codeの開発環境
    - Pythonプロジェクト（または対応言語）
    - 品質ツール（ruff、mypy、pytestなど）の設定
- **Decision Triggers**:
    - Issue駆動開発のフローを手動で行うのが煩雑になってきた
    - PR前の品質チェックを忘れることが多い
    - TDDを徹底したいが、手動では遵守が難しい
    - 複数のIssueを並行して進める必要が出てきた
    - コードレビューの自動化を検討している
- **Failure Modes / Risks**:
    - 初期設定が不適切だと品質ツールが正しく動作しない
    - ワークツリーの管理不足によるディスク容量の圧迫
    - 自動化に依存しすぎて、手動での柔軟な対応が難しくなる可能性
    - Claude Codeのバージョンや公式プラグインの変更による互換性問題
- **Operational Cost**:
    - 初回セットアップ：workflow-config.jsonの設定、品質ツールのインストール
    - 学習コスト：各スラッシュコマンドの使い方とワークフローの理解
    - 維持コスト：品質ツールのバージョン管理、設定ファイルのメンテナンス
