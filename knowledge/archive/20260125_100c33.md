# Claude Agent Skills のベストプラクティス

## TL;DR & Key Conclusions
この記事は、Anthropic公式ドキュメント「Skill authoring best practices」の日本語解説です。Agent Skillsを効果的に構築するための原則として、簡潔性の重視、タスク性質に応じた自由度調整、段階的情報開示、フィードバックループの活用を解説しています。

- **簡潔さが最優先**: コンテキストウィンドウは複数スキルと会話履歴で共有される「公共財」。冗長な説明や一般知識の再説明は不要
- **自由度を調整**: 複数アプローチが有効なタスクには高い自由度、デリケートな操作には低い自由度を設定
- **Progressive Disclosure**: SKILL.mdは目次として機能し、詳細は別ファイルに分割。起動時は必要最小限のみロード
- **フィードバックループ**: 検証→修正→再検証のループで出力品質を大幅向上
- **段階的情報開示は1階層まで**: 参照の深さを制限して複雑性を管理
- **Claudeと反復開発**: Claude自身をパートナーとしてスキルを開発・改善するプロセスが推奨される
- **命名規則**: 動名詞形（`processing-pdfs`）が推奨。最大64文字、小文字・数字・ハイフンのみ

## Quickstart
1. **スキル設計**: タスクの再利用パターンを特定し、必要な自由度レベルを決定
2. **SKILL.md作成**: 必須フィールド（name, description）を含む簡潔な目次を作成
3. **段階的情報開示**: 詳細情報を別ファイルに分割（1階層まで）
4. **フィードバックループ追加**: 検証ステップを明示し、修正→再検証の流れを組み込む
5. **反復テスト**: 別のClaude会話で実際に使用してフィードバックを収集
6. **簡潔性確認**: 冗長な説明や一般知識の説明を削除

## Context & Claims (Claim-Evidence-Caveat)

- **Claim**: 簡潔さが最も重要な原則である
    - **Evidence**: コンテキストウィンドウは複数スキルと会話履歴で共有される「公共財」。良い例は約50トークン、悪い例は約150トークン（3倍の差）
    - **Caveat**: Claudeが既に知っている一般知識（PDFの概念など）は説明不要だが、プロジェクト固有の文脈は必要

- **Claim**: タスクの性質に応じて自由度を調整すべき
    - **Evidence**: 「橋と野原」の比喩で3段階に分類。コードレビューは高自由度、データベースマイグレーションは低自由度が適切
    - **Caveat**: 高自由度タスクでも、無制限な選択肢は混乱を招く。デフォルト推奨を明示し、例外ケースのみ代替案を提示

- **Claim**: Progressive Disclosure（段階的情報開示）でコンテキスト効率を最大化
    - **Evidence**: SKILL.mdは起動時に全スキルで読み込まれるが、詳細ファイルは実行時のみロード。参照は1階層までに制限
    - **Caveat**: 深いネスト（参照の参照）は複雑性を増し、メンテナンスを困難にする

- **Claim**: フィードバックループが出力品質を大幅に向上させる
    - **Evidence**: 検証→修正→再検証のステップを明示することで、単一パス処理より高品質な結果を実現
    - **Caveat**: すべてのタスクにフィードバックループが必要なわけではない。シンプルな操作には過剰

- **Claim**: Claudeを開発パートナーとして活用すべき
    - **Evidence**: 推奨プロセスは7ステップ：①タスク完了 ②パターン特定 ③Skill化依頼 ④簡潔性確認 ⑤別会話でテスト ⑥改善 ⑦再テスト
    - **Caveat**: Claude生成のスキルも必ず人間がレビューし、簡潔性や構造を確認する必要がある

- **Claim**: 命名規則と必須フィールドを遵守すべき
    - **Evidence**: `name`は最大64文字（小文字・数字・ハイフン）、`description`は最大1024文字で三人称必須。動名詞形が推奨
    - **Caveat**: 既存スキルとの一貫性も重要。プロジェクト独自の命名規則がある場合は調整が必要

- **Claim**: アンチパターンを避けることで互換性と保守性が向上
    - **Evidence**: Windowsパス（`C:\`）は避け、Unix形式（`scripts/helper.py`）を使用。時間依存情報は`<details>`タグで隠す
    - **Caveat**: プラットフォーム固有の操作が必要な場合は、条件分岐で対応する必要がある

## Normalized Conditions

- **Use Cases**:
    - Agent Skillsを新規作成する場合
    - 既存スキルのリファクタリングや品質改善
    - 再利用可能なタスクパターンを特定した場合
    - コンテキストウィンドウの効率化が必要な場合
    - 複数人でスキル開発を行う場合（命名規則統一のため）

- **Anti-Cases**:
    - 一度きりのタスク（再利用性がない）
    - プロジェクト固有すぎて汎用化できない処理
    - 既存スキルで十分対応できる場合
    - コンテキストウィンドウに余裕がある単純なタスク

- **Decision Triggers**:
    - 同じタスクを3回以上実行した場合
    - コンテキストウィンドウが逼迫してきた場合
    - スキルの保守性や可読性に問題が生じた場合
    - チーム開発でスキル品質のばらつきが問題になった場合
    - 公式ドキュメントのベストプラクティスに準拠させたい場合
