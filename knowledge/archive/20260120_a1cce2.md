# Claude Code のサブエージェントで GitHub Issue を並列処理する方法 ── CCPM という解答

## TL;DR & Key Conclusions
Claude Codeで複数のGitHub Issueを並列処理する際、コンテキストが混ざって開発が破綻する問題に対する解決策としてCCPMが有効。CCPM（Claude Code PM）は6,000スター超のOSSで、Issue単位でコンテキストを完全分離し、Git worktreesを活用した並列エージェント実行を実現する。

- **コンテキスト分離の実現**: Issue単位で専用エージェントが起動し、他タスクと混ざらない独立したコンテキストを保持
- **並列実行による高速化**: 1つのIssueを複数のワークストリームに分解し、最大3倍の開発速度向上が可能
- **トレーサビリティの確保**: PRD → Epic → Task → GitHub Issue → Codeという明確なトレースパスで仕様から実装まで追跡可能
- **2分で導入可能**: 既存プロジェクトへの導入が極めて簡単で、.claudeディレクトリをコピーするだけで完了
- **GitHub Issues中心の設計**: 複数のClaudeインスタンスがプロジェクトで同時稼働でき、人間が進捗をリアルタイム確認可能
- **vibe codingからの脱却**: すべてのコードが仕様にトレース可能な「spec-driven development」への移行を支援
- **チームコラボレーション**: AIと人間が同じIssue上で協調作業でき、作業の引き継ぎもスムーズ
- **他ツールとの比較優位**: Claude Flowやmyclaudeといった代替ツールに比べ、セットアップの簡単さとドキュメント充実度で優位

## Quickstart
1. **インストール**: 既存プロジェクトに.claudeディレクトリをコピーする（約2分）
2. **初期化**: `/pm:init`コマンドでGitHub CLIの認証と必要ディレクトリを自動生成
3. **PRD作成**: `/pm:prd-new [機能名]`で製品要件定義を対話形式で作成
4. **Epic変換**: `/pm:prd-parse [機能名]`でPRDを技術実装計画に変換
5. **タスク分解**: `/pm:epic-oneshot [機能名]`でEpicをタスクに分解しGitHub Issuesに登録
6. **Issue処理開始**: `/pm:issue-start [Issue番号]`で専用エージェントを起動
7. **進捗同期**: `/pm:issue-sync [Issue番号]`でGitHub Issueに進捗を記録
8. **次タスク取得**: `/pm:next`で優先度に基づき次のIssueを提示

## Context & Claims (Claim-Evidence-Caveat)
- **Claim**: Claude Codeの複数タスク同時処理はコンテキストの混同で破綻する
    - **Evidence**: 1つのセッションで複数Issueを扱うと、認証機能の実装中にAPIエンドポイントやUIコンポーネントの議論が混入し、やり直しが発生する
    - **Caveat**: /compactや/clearで対処可能だが、頻繁な使用が必要で効率が悪い

- **Claim**: CCPMはGitHub IssuesとGit worktreesを組み合わせたプロジェクト管理システム
    - **Evidence**: 6,000スター超、実運用での実績が十分にあるOSS
    - **Caveat**: GitHub Issuesを必須とするため、Issueを使わない開発フローには不向き

- **Claim**: CCPMは並列実行により開発速度を最大3倍向上させる
    - **Evidence**: 1 Issue = 複数の並列ワークストリームとして扱い、データベース、サービスレイヤー、API、UI、テストを同時並行で開発可能
    - **Caveat**: 速度向上は機能の規模と複雑さに依存し、すべてのケースで3倍になるわけではない

- **Claim**: すべてのコードが仕様にトレース可能
    - **Evidence**: PRD → Epic → Task → GitHub Issue → Code → Commitという設計思想で、なぜこのコードがこうなっているのかをPRDまで遡って確認できる
    - **Caveat**: トレーサビリティの恩恵を受けるには、PRDとEpicを適切に管理する必要がある

- **Claim**: GitHub Issuesをデータベースとして採用することでチームコラボレーションが強化される
    - **Evidence**: 複数のClaudeインスタンスが同一プロジェクトで同時稼働可能、人間がIssueコメントでAIの進捗をリアルタイム確認、AIと人間の作業引き継ぎが可能
    - **Caveat**: GitHub Issuesへの依存が強く、プライベートな環境やオフライン開発には不向き

- **Claim**: Claude Flowはより高度だがセットアップが複雑
    - **Evidence**: 175以上のMCPツール、Swarmベースのエージェント管理、学習機能付きルーティングを提供
    - **Caveat**: 学習コストが高く、大規模プロジェクトや高度なカスタマイズが必要な場合のみ検討すべき

- **Claim**: myclaudeは軽量だが構造化ワークフローが弱い
    - **Evidence**: `/gh-issue-implement 123`のようなシンプルなコマンドでIssueから直接実装可能
    - **Caveat**: CCPMほどの構造化されたワークフローは持たず、トレーサビリティや並列実行機能で劣る

## Code Snippets
- **インストールスクリプト**: プロジェクトルートで実行し、.claudeディレクトリを導入する1行コマンド
- **既存プロジェクトへのマージ**: 既存の.claudeディレクトリがある場合に内容をマージするコマンド
- **並列実行コマンド群**: Issue分析、並列実行開始、マージまでの一連のコマンド

## Normalized Conditions
- **Use Cases**:
    - 複数のGitHub Issueを並列で処理したい開発チーム
    - コンテキストスイッチを削減し、集中して開発したい個人開発者
    - 仕様から実装までのトレーサビリティを確保したいプロジェクト
    - AIと人間が協調してタスクを進めるハイブリッド開発体制
    - vibe codingから脱却し、spec-driven developmentに移行したいチーム
    - Claude Codeで複数タスクを同時進行させたいが、/compactや/clearを頻繁に使う状況
- **Anti-Cases**:
    - GitHub Issuesを使わない開発フロー
    - 単一のシンプルなタスクのみを扱う場合
    - プロトタイピングや小規模な単発タスク
    - オフライン環境での開発
    - チーム内でのコラボレーションが不要な個人プロジェクト
- **Prerequisites**:
    - GitHub CLIのインストールと認証
    - Git worktreesの理解（ツールが自動処理するため必須ではない）
    - GitHub Issuesを活用した開発フロー
    - Claude Codeの基本的な使用経験
- **Decision Triggers**:
    - Claude Codeで複数Issueを同時に扱い始めた時
    - /compactや/clearの使用頻度が増えてきた時
    - コンテキストの混同によるやり直しが頻発した時
    - 開発チームでClaude Codeを共同利用し始めた時
    - 仕様と実装のトレーサビリティを求められた時
- **Failure Modes / Risks**:
    - GitHub Issuesへの過度な依存によるオフライン作業の制限
    - worktreesの管理ミスによるブランチの混乱
    - 複雑なEpic分解によるオーバーヘッド増加
    - PRDやEpicの品質が低い場合、トレーサビリティの価値が低下
    - 並列実行時のマージコンフリクトのリスク
- **Operational Cost**:
    - 初回セットアップで約2分
    - PRDとEpicの作成・管理コスト
    - GitHub Issuesへの進捗同期コスト
    - Git worktreesの管理とマージ作業
    - 各Issueごとのエージェント起動コスト
