# GitHub Projects (v2) で個人の Issue/PR 管理をエンジニアリングする

## TL;DR & Key Conclusions

GitHub Projects v2を使った個人のIssue/PR管理を完全自動化し、認知負荷とコストを削減した事例。フィールド設計をシンプル化し、Go言語とGitHub APIで自動化ツールを実装。平日営業時間のみ実行することで月額コストを2,000円から750円に削減しながら、50個以上の放置Issue/PRを発掘・整理できた。

- **課題**: Issue/PRの追加漏れ、手動ステータス更新の負担、フィールド肥大化、認知負荷の増大
- **解決策**: 2次元のシンプルなフィールド設計（Category × Status）と自動化ルールの機械判定可能化
- **実装**: Go言語でGitHub API v3を使用、Cloud Run + Cloud Schedulerで3分ごとに実行
- **成果**: 全自動化により快適な運用を実現、フィールド数を大幅削減、放置Issue/PRの発掘
- **コスト最適化**: 実行時間帯を平日営業時間に制限して月額750円に削減（元2,000円）
- **公開リポジトリ**: https://github.com/igsr5/github-project-automation-prototype

## Quickstart

GitHub Projects v2で個人のIssue/PR管理を自動化する手順：

1. **フィールド設計**: CategoryフィールドとStatusフィールドの2つのみを作成
   - Category: `Issues`, `PRs`, `Review PRs`
   - Status: `In Progress`, `In Pending`, `Complete`
2. **自動化ロジックの実装**: Go言語でGitHub API v3を使用してプロジェクト状態をチェック・更新
3. **デプロイ**: Google Cloud RunにデプロイしCloud Schedulerで定期実行（平日9-18時、3分間隔を推奨）
4. **レート制限対策**: 実行頻度を調整（3分間隔が目安）
5. **コスト最適化**: 実行時間帯を営業時間のみに制限

## Context & Claims (Claim-Evidence-Caveat)

- **Claim**: GitHub標準のWorkflow機能では複雑な自動化要件を満たせない
  - **Evidence**: 著者が求める自動化ルール（PRマージ→Complete、レビューリクエスト受信→Review PRs/In Progress等）は標準機能では実現できず、カスタム実装が必要だった
  - **Caveat**: 標準機能は今後拡充される可能性があり、将来的にはカスタム実装が不要になる可能性がある

- **Claim**: フィールド設計をシンプル化（2次元）することで認知負荷を劇的に削減できる
  - **Evidence**: 複数のカスタムフィールドを「Category（Issues/PRs/Review PRs）」と「Status（In Progress/In Pending/Complete）」の2つに集約し、8つの状態を管理
  - **Caveat**: 個人の作業スタイルによって最適なフィールド設計は異なる。チーム運用では別の設計が必要

- **Claim**: 機械判定可能なルールとして状態遷移を設計することで完全自動化が可能
  - **Evidence**: 「PRがマージされる→Complete」「レビューリクエスト受信→Review PRs/In Progress」等、明確な条件で自動判定
  - **Caveat**: 例外ケースや複雑な判断が必要な場合は手動介入が必要になる可能性がある

- **Claim**: 実行時間帯を制限することで大幅なコスト削減が可能
  - **Evidence**: 24時間フル稼働（月2,000円）から平日9-18時のみ実行（月750円）に変更し、約62%のコスト削減を実現
  - **Caveat**: 営業時間外の更新は翌営業日まで反映されない。リアルタイム性が求められる場合は不向き

- **Claim**: GitHub API v3とGo言語の組み合わせで安定した自動化ツールを構築できる
  - **Evidence**: 著者が実装したツールはCloud Run + Cloud Schedulerで安定稼働し、レート制限にも対応
  - **Caveat**: GitHub APIのレート制限があるため、実行頻度の調整が必要（3分間隔を推奨）

- **Claim**: 自動化により放置されていたIssue/PRを発掘できる
  - **Evidence**: 運用開始後、50個弱の放置Issue/PRが発見され整理された
  - **Caveat**: 発掘されたIssue/PRの整理には人間の判断が必要

## Normalized Conditions

- **Use Cases**:
  - 個人での大量のIssue/PR管理（複数リポジトリにまたがる場合も有効）
  - Issue/PRの追加漏れや更新漏れを防ぎたい場合
  - 手動でのステータス管理による認知負荷を減らしたい場合
  - シンプルなフィールド設計で運用を楽にしたい場合
  - 放置されたIssue/PRを発掘・整理したい場合

- **Anti-Cases**:
  - リアルタイムな更新が必須の場合（営業時間外の更新は翌営業日まで反映されない）
  - チーム運用で複雑なワークフローが必要な場合（個人向け設計）
  - GitHub標準機能で十分な簡易的な管理の場合

- **Decision Triggers**:
  - Issue/PRの管理が手動で追いつかなくなった場合
  - フィールド数が増えすぎて管理が煩雑になった場合
  - 複数リポジトリのIssue/PRを一元管理したい場合
  - 月額数百円のコストを許容できる場合
