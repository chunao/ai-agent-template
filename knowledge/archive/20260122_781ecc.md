# Claude CodeとCodexの連携をMCPからSkillに変えたら体験が劇的に改善した

## TL;DR & Key Conclusions
Claude CodeとCodex CLIの連携において、MCPサーバー方式からSkillベースの実装に移行することで、進捗の可視性・デバッグ性・操作性が劇的に向上する。特に長時間実行タスクではSkillベースが圧倒的に優れている。

- **MCPの致命的欠点**: ツール実行中の進捗が完全に不可視で、数十分〜1時間以上の無応答状態になることがある
- **Skillの決定的優位性**: ターミナル出力がリアルタイムで確認でき、状況把握と中断判断が容易
- **呼び出し方法の改善**: プロンプトでの指示から `/codex` コマンドによる明示的な呼び出しに変更
- **デバッグ性の向上**: MCPのブラックボックス状態からエラー出力が直接見える透明な状態へ
- **推奨される用途**: コードレビュー、バグ調査、設計相談、文言検討、困難な問題の調査

## Quickstart
1. **Skillファイルの作成**: Claude Codeのskillsディレクトリにcodex用のskillを作成
2. **基本コマンドの設定**: `codex exec --full-auto --sandbox read-only --cd <project_directory> "<request>"` を実行コマンドとして定義
3. **呼び出し方法**: `/codex` コマンドでSkillを呼び出し

## Context & Claims (Claim-Evidence-Caveat)
- **Claim**: MCPサーバー方式は長時間タスクに適さない
    - **Evidence**: 複雑なタスクで数十分〜1時間以上の無応答状態が発生。進捗表示がなく、タイムアウトかどうかも判断困難
    - **Caveat**: 短時間で完了するシンプルなタスクではMCPでも問題ない可能性がある

- **Claim**: Skillベース実装により開発体験が劇的に改善
    - **Evidence**: リアルタイムの進捗表示、エラー出力の可視化、中断判断の容易さを実現
    - **Caveat**: Skill実装には初期セットアップのコストがかかる

- **Claim**: `/codex`コマンド呼び出しがプロンプト指示より優れている
    - **Evidence**: 明示的なコマンドで確実にCodexを呼び出せる。プロンプトでは意図通りに動作しないことがある
    - **Caveat**: コマンド名を覚える必要がある

## Code Snippets
- **Codex実行コマンド**: `codex exec --full-auto --sandbox read-only --cd <project_directory> "<request>"` - Skillから Codex CLIを呼び出す際の基本コマンド。full-autoで自動実行、read-onlyサンドボックスで安全に実行

## Normalized Conditions
- **Use Cases**:
    - 長時間実行が予想されるタスク（コードレビュー、バグ調査など）
    - 進捗状況を把握しながら作業したい場合
    - デバッグ情報が必要な複雑なタスク
    - Claude CodeとCodexを組み合わせた開発ワークフロー
- **Anti-Cases**:
    - 数秒で完了する単純なタスク
    - MCPの統一的なインターフェースが必要な場合
    - Skill実装のセットアップコストをかけられない場合
- **Prerequisites**:
    - Claude Code CLI
    - Codex CLI
    - Skillの作成方法の理解
- **Decision Triggers**:
    - MCPでCodexを使っていて進捗が見えずストレスを感じたとき
    - 長時間タスクでタイムアウトしたかどうか判断できないとき
    - Codex実行中のデバッグ情報が必要になったとき
